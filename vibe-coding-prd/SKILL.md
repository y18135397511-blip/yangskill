# Vibe Coding PRD 生成器

帮助 AI PM 从模糊想法快速生成可直接执行的 PRD。不整虚的，就五个问题 + 一页纸文档。

## 触发条件

当用户有以下表达时，触发此 Skill：
- "帮我写个PRD"
- "我想做一个XX"
- "帮我做个YYY"
- "做个ZZZ工具"
- "帮我梳理需求"
- "我要vibe code一个产品"
- 任何关于快速创建产品需求文档的请求

---

## 工作流程

### 第一步：判断用户状态

**先开口**：

> 看你的描述，这个想法目前比较清晰了，我确认几个关键点就能出PRD。
> 
> 或者：
> 
> 你的想法有点多，我先帮你聚焦到一个核心点上来。

**判断标准**：
- 想法很模糊（只有一句话或一个词）→ 进入引导模式
- 需求已经比较清晰（能说出给谁用、做什么）→ 进入确认模式

直接告诉用户你现在属于哪种情况，然后开始。

---

### 第二步：需求澄清（5个问题）

每次只问一个，等回答了再问下一个。不要一次性抛出所有问题。

#### 问题1：为什么要做这个？

**话术**：
> 先问一下：你为什么想做这个？或者说，这个产品解决什么问题？

**为什么问这个**：如果"为什么"说不清楚，后面写的所有功能都是空中楼阁。很多产品做到一半发现没人用，就是一开始没想清楚为什么做。

**如果用户回答不上来**，可以追问：
- 现在让你不爽的一件事是什么？
- 你是看到别人做了什么产品，你想做个类似的？
- 还是你自己有什么需求市面上没满足？

---

#### 问题2：给谁用？

**话术**：
> 这个产品给谁用的？

**追问技巧**：不接受"所有人"这种回答。

**如果用户说"所有人"**，继续问：
> 那谁最可能用？你身边谁会用这个？
> 
> 或者：
> 
> 换个问法：谁用了会特别爽？谁用了会觉得没用？

**为什么问这个**：产品做给所有人用 = 没人用。没有明确用户画像的PRD最后都会变成"加一点功能、加一点功能"，然后项目爆炸。

---

#### 问题3：核心功能是什么？

**话术**：
> 用户用这个产品，最核心的一个功能是什么？如果只能选一件事的话。

**追问技巧**：用户可能会说很多功能，这时候要帮忙砍。

> 听起来你想做 ABCD 四个功能，咱们先砍到只能选一个的话，你觉得哪个是最重要的？

**为什么问这个**：Vibe coding 的精髓就是先做一个功能做透，而不是做十个功能都是半成品。

---

#### 问题4：明确不做什么？

**话术**：
> 这个产品有什么不做的？或者说，第一版先不包含什么？

**为什么问这个**：没有边界的PRD会无限膨胀，最后做不完。这个问题比"做什么"更重要。

**常见边界**：
- 不需要用户登录
- 不需要支付功能
- 不需要后台管理系统
- 不需要分享到社交媒体
- 第一版只做 web 端

---

#### 问题5：怎么算成功？

**话术**：
> 怎么就算这个产品做成了？有什么验收标准？

**为什么问这个**：没有验收标准的PRD就是许愿清单，做完了也不知道算不算完成。

**常见标准**：
- 用户能完成 XX 操作
- 页面加载时间 < X 秒
- 能正常在手机上打开
- AI 生成的内容满足基本要求

---

### 第三步：技术方案建议

根据前面的回答自动判断产品类型，推荐技术方案。

**判断逻辑**：

| 产品类型 | 技术方案 | 为什么 |
|---------|---------|-------|
| 纯展示类（导航站、简单主页、无交互） | HTML + Tailwind + 静态部署 | 最简单，不需要服务器，直接部署到 Cloudflare Pages 或 Vercel 就能跑 |
| 有AI交互（内容生成、对话、智能处理） | Next.js + AI API + Vercel | Vercel 部署方便，Next.js 生态好，AI API 一行代码就能接入 |
| 需要存数据（用户记录、历史、持久化） | Next.js + Supabase + Vercel | Supabase 帮你搞定后端和数据库，不用自己搭服务器，免费额度够个人项目用 |
| 不确定 | Next.js + Tailwind + Vercel | 最主流的全栈方案，生态最丰富，做什么都能扩展 |

**输出话术**：
> 根据你说的这些，我建议用 XX 技术栈，原因是……如果你有其他偏好也可以告诉我。

---

### 第四步：生成一页纸PRD

把前面的信息整合，输出结构化文档：

```markdown
# [产品名称] PRD

## 产品定位
一句话说清楚这个产品是什么：
> 给 XX 用户用的 YY 产品，帮他们解决 ZZ 问题。

## 目标用户
- [用户群体描述，2-3句话]
- 典型场景：用户在 XX 情况下会用这个产品

## 核心功能（最多3个）

### 功能1：[功能名]
- **用户操作**：用户打开页面 → 输入/选择 XX → 点击 YY
- **预期结果**：系统展示 ZZ / 生成 AA / 保存 BB
- **验收标准**：
  - [ ] 用户能顺利完成操作
  - [ ] 操作后有明确反馈
  - [ ] 异常情况有错误提示

### 功能2：[功能名]
（同上格式）

### 功能3：[功能名]
（同上格式）

## 明确不做（本版本排除）
- [ ] 不包含用户登录/注册
- [ ] 不支持移动端
- [ ] 不包含分享功能
- [ ] 不做数据持久化
- [ …… ]

## 技术方案
- **前端**：Next.js + Tailwind
- **后端**：Vercel Serverless Functions
- **AI**：使用 XX API
- **数据**：本地存储 / Supabase
- **部署**：Vercel
- **选型理由**：因为 YY 需求，所以选 XX 技术

## 产品风格
- **UI风格**：简洁 / 卡片式 / 极简 / 工具感
- **主色调**：XX 色系（#XXXXXX）
- **语言**：中文 / 英文 / 双语
- **整体氛围**：专业 / 轻快 / 极客 / 温暖

## 给 AI 的执行指令

这是最关键的部分，直接告诉 AI 怎么执行：

1. **初始化**：用 `npx create-next-app@latest` 初始化项目，选择 TypeScript + Tailwind
2. **页面结构**：只需要 X 个页面——首页 + YY 页面
3. **核心功能实现顺序**：先做 ZZ 功能，再做 AA 功能
4. **特别注意**：
   - 不要做用户登录
   - 优先保证移动端体验
   - 页面加载不能超过 X 秒
5. **验收**：完成后自己测试一遍，确保用户能流畅完成 YY 操作
6. **部署**：直接部署到 Vercel，不需要额外配置
```

---

### 第五步：质量自检

生成后对照清单自动检查，每一项都要确认：

- [ ] **一句话说清楚**：产品定位那一行，如果读的人不能立刻明白这个产品是做什么的，就是失败的
- [ ] **MVP 可完成**：一个下午能做完吗？如果不能，说明功能砍得还不够狠
- [ ] **边界清晰**："明确不做"列了吗？至少列 3 条
- [ ] **验收标准**：每个功能都有验收标准吗？每个功能至少 2 条
- [ ] **技术匹配**：选的技术方案能支撑这个需求吗？比如纯展示站不需要 Supabase
- [ ] **可直接执行**：把这份 PRD 给 AI，它能立刻开始写代码吗？还需要追问吗？

**如果任何一项不通过**，主动说：
> 我检查了一遍，发现 XX 这块不太够，我调整一下……

然后主动修改。

---

## 输出要求

1. **一页纸原则**：PRD 控制在一页内，不要展开太多细节
2. **可直接执行**：文档要详细到 AI 能直接理解并执行，不需要再追问细节
3. **语气**：像有经验的 AI PM 朋友在帮你梳理思路，不要太正式
4. **技术仅供参考**：最终技术选型让 AI 根据实际情况决定

---

## 常见问题处理

**用户想法太多/太模糊**：
> 听起来你想做的有点多，咱们先聚焦到一个最核心的点。你最想先做的是哪个？

**用户说"所有人"都用**：
> 那谁用了会最爽？比如 程序员？运营？大学生？先想一个具体的人。

**用户说不清楚为什么做**：
> 换个问法：你自己有什么需求市面上没满足？或者你看到什么产品觉得可以改进？

**用户功能列了一堆**：
> 咱们先砍到只能做一个，做深做透比做全更重要。你觉得哪个是核心中的核心？

**用户不确定技术选型**：
> 我帮你选了一个最合适的方案，如果你有特定偏好也可以说。

---

生成完 PRD 后，提醒用户：
> PRD 好了。你把这个发给 AI（Claude Code、Cursor 什么的），它就能开始写了。
